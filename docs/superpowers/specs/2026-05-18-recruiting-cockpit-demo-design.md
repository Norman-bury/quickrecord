# 招聘提效管理驾驶舱 Demo 设计

日期：2026-05-18

## 背景

本 demo 对应“招聘提效-场景一”：HR 在企业微信、群聊、腾讯在线文档等渠道中手动记录招聘过程，导致效率低、数据遗漏、管理层难以及时判断招聘进展。目标是在最小可落地范围内，用真实 OpenAI API 将非结构化招聘沟通内容转成候选人数据，并在单页管理驾驶舱中展示招聘效率和风险。

## 目标用户

主要用户是企业管理层。界面要优先回答：

- 当前招聘整体推进是否健康。
- 哪些岗位、候选人、渠道存在风险。
- AI 自动记录相比人工记录节省了多少时间。
- 数据是否已经从聊天内容同步为可分析的招聘流水。

HR 录入体验是次要目标，只保留 demo 所需的导入、提取、修正和阶段更新能力。

## 产品范围

### 包含

- 单页 Web App 管理驾驶舱。
- 内置中文招聘沟通样例，打开即可演示。
- 粘贴企业微信、微信群或腾讯文档风格的招聘记录。
- 调用 OpenAI API，将文本结构化为候选人数据。
- 候选人列表和招聘阶段跟踪。
- 管理层指标：招聘漏斗、渠道表现、风险提醒、平均推进天数、节省工时。
- API key 缺失或调用失败时给出明确错误。

### 不包含

- 真实企业微信 API 接入。
- 真实腾讯文档 API 接入。
- 登录、权限、组织架构、多租户。
- 持久化数据库。
- 正式生产环境部署。

## 信息架构

页面采用单页三块结构：

1. 顶部管理概览：展示总候选人、活跃岗位、面试转化率、风险数量、估算节省工时。
2. 左侧数据导入区：展示样例文本、粘贴文本、触发 AI 提取、查看提取状态和错误。
3. 主体驾驶舱：候选人流水表、招聘漏斗、渠道效率、风险提醒和阶段操作。

布局应偏企业后台工具风格：信息密度高、视觉克制、强调扫描和对比，不做营销式落地页。

## 数据模型

候选人字段：

- `id`：本地唯一 ID。
- `name`：候选人姓名。
- `role`：应聘岗位。
- `source`：渠道，例如 Boss 直聘、内推、猎头、官网投递。
- `stage`：阶段，限定为 `初筛`、`沟通中`、`待面试`、`已面试`、`待Offer`、`已入职`、`已淘汰`。
- `owner`：负责人。
- `interviewTime`：面试时间，允许为空。
- `lastContact`：最近沟通时间或日期描述。
- `risk`：风险标签，例如 `待反馈`、`时间冲突`、`薪资风险`、`信息缺失`。
- `summary`：一句话摘要。
- `confidence`：AI 提取置信度，0 到 1。

派生指标：

- 招聘漏斗：按阶段统计候选人数。
- 渠道效率：按 `source` 统计候选人数、进入面试人数、Offer/入职人数。
- 风险数量：存在 `risk` 的候选人数。
- 节省工时：按每条候选人记录节省 6 分钟估算。
- 平均推进天数：demo 中根据阶段和样例日期做近似展示，不作为真实统计。

## OpenAI 调用

后端提供 `POST /api/extract`。

请求：

```json
{
  "text": "招聘沟通记录..."
}
```

响应：

```json
{
  "candidates": [
    {
      "name": "张三",
      "role": "前端工程师",
      "source": "内推",
      "stage": "待面试",
      "owner": "Lisa",
      "interviewTime": "周三 15:00",
      "lastContact": "今天",
      "risk": "待确认面试时间",
      "summary": "已通过初筛，等待确认面试",
      "confidence": 0.86
    }
  ]
}
```

服务端使用 `OPENAI_API_KEY` 环境变量。模型返回必须约束为 JSON；服务端负责校验和归一化字段。如果未配置 API key，接口返回 500 和明确错误信息；前端显示“未配置 OPENAI_API_KEY”，不伪造成功。

## 技术方案

采用方案 1：本地单页 Web App + Node/Express API。

- 前端：Vite + React + TypeScript。
- 样式：CSS Modules 或普通 CSS，避免引入重型 UI 框架。
- 图标：使用 `lucide-react`。
- 后端：Node.js + Express。
- OpenAI：官方 `openai` Node SDK。
- 开发运行：一个命令启动前端和后端，或分别运行 `npm run dev` 与 `npm run server`。

项目可以不使用数据库，候选人数据保存在前端状态中。刷新页面后回到内置样例状态，这符合 demo 的可重复演示目标。

## 交互流程

1. 用户打开页面，看到默认样例文本和一组已存在样例候选人，用于无 API key 时预览界面。
2. 用户粘贴招聘沟通记录，点击“AI 提取”。
3. 前端调用 `POST /api/extract`。
4. 后端调用 OpenAI，将文本转成候选人 JSON。
5. 前端将提取结果合并到候选人表。
6. 管理层指标、漏斗、渠道效率和风险提醒自动刷新。
7. 用户可以在候选人表里修改阶段，指标随之更新。

## 错误处理

- 空文本：前端阻止提交并提示需要输入招聘记录。
- API key 缺失：后端返回明确错误，前端显示配置提示。
- OpenAI 返回非 JSON：后端返回解析失败，前端提示“AI 返回格式异常，请重试或调整输入”。
- 网络失败：前端显示请求失败，并保留当前候选人数据。
- 字段缺失：服务端用默认值归一化，例如渠道为“未识别”、阶段为“初筛”、置信度为 `0.5`。

## 验收标准

- 页面打开后无需登录即可看到完整驾驶舱。
- 设置 `OPENAI_API_KEY` 后，粘贴中文招聘记录能调用真实 OpenAI API 并生成候选人数据。
- 未设置 `OPENAI_API_KEY` 时，页面仍可展示样例驾驶舱，并在点击 AI 提取时给出明确配置错误。
- 候选人阶段变更后，顶部指标、漏斗和渠道分析同步变化。
- 管理层能在一分钟内看懂：招聘进展、渠道效果、风险项、自动化节省工时。
- 本地运行命令和环境变量说明写入 README。

## 测试策略

- 单元测试：校验候选人归一化、指标计算、阶段统计、渠道统计。
- API 测试：校验空文本、缺少 API key、模拟合法 OpenAI JSON 的响应处理。
- 手动验证：启动本地服务，在浏览器中完成样例导入、真实提取、阶段修改和错误提示验证。
